# Advanced Studio Template System

This document preserves the current implementation-specific documentation for Advanced Studio behavior templates. It is intentionally separate from `docs/advanced-studio-architecture.md`, which remains the long-lived system architecture reference.

Observed fact:

The current Template system is a scene behavior/composition recipe map. It is not a first-class scene template system in the schema.

## Current Template System

The current Template system exists in `studio/src/AdvancedStudioApp.tsx`. It is a local UI and orchestration system inside the Advanced Studio app, not a new core schema or renderer.

### Template model

`SceneCompositionRecipe` currently contains:

```ts
type SceneCompositionRecipe = {
  cameraPath: NonNullable<AdvancedStudioScene["cameraPath"]>;
  transitionIn?: AdvancedStudioScene["transitionIn"];
  transitionOut?: AdvancedStudioScene["transitionOut"];
  boardAnimation?: AnimationPreset;
  boardProject?: StudioProject;
  designId?: string;
};
```

`BehaviorTemplateId` is `"template-1" | "template-2"`.

The Transition tab shows two buttons:

- Template 1
- Template 2

Button selection is data-grounded. `sceneMatchesComposition` compares the selected scene against a recipe by checking camera preset, transition values, Board animation, Board project JSON equality, and infographic design ID.

Clicking a template calls `applyCompositionRecipe` for the currently selected scene only.

Template application now runs provider-design compatibility validation before applying an infographic design change. If a recipe would pair the current scene content with an incompatible provider design, Advanced Studio preserves the current scene and reports the compatibility reason instead of letting the provider renderer fail.

## Template 1

Template 1 is the original authored behavior and composition from `advancedStudioDefaultProject`.

It is generated by:

```text
advancedStudioDefaultProject.scenes
  -> compositionFromScene(scene)
  -> template1CompositionBySceneId
```

Template 1 changes nothing by itself. It is a reliable restoration map for the original scene values.

Template 1 can restore:

- camera path
- transition in
- transition out
- Board animation
- Board project
- infographic design ID

Template 1 cannot restore fields that are not part of `SceneCompositionRecipe`. It does not include scene order, scene duration, text copy, rows, nodes, edges, controls, title, format, audio, or export settings.

Owners modified when Template 1 is applied:

- Advanced scene `cameraPath`
- Advanced scene `transitionIn`
- Advanced scene `transitionOut`
- Board scene `content.project`
- Board scene `content.animation`
- Infographic scene `content.designId`

Owners untouched when Template 1 is applied:

- Renderers
- Provider factories
- Provider data rows/nodes/edges
- Scene duration
- Scene order
- Timeline helper logic
- Camera transform implementation
- Transition opacity implementation
- Board semantic motion resolver
- Provider animation wrapper implementation

Template 2 cannot currently change:

- camera implementation
- transition implementation

Template 2 modifies owners by selecting/configuring existing owners. It does not define a new renderer, new provider, new scene type, new transition system, new camera system, or new animation engine.

## Template Ownership

Template 1 and Template 2 are currently applied by `applyCompositionRecipe`.

`applyCompositionRecipe` modifies only these scene fields:

- `scene.cameraPath`
- `scene.transitionIn`
- `scene.transitionOut`
- for Board scenes: `scene.content.project`
- for Board scenes: `scene.content.animation`
- for infographic scenes: `scene.content.designId`

`applyCompositionRecipe` does not modify:

- scene order
- scene duration
- scene title
- infographic rows
- infographic nodes
- infographic edges
- infographic controls
- renderer implementation
- camera transform implementation
- transition opacity implementation
- Board semantic motion implementation
- provider animation implementation

The template system selects existing owners. It does not own rendering behavior directly.

Compatibility ownership:

- Provider designs declare capability metadata through `AntVStudioDesign.capabilities`.
- `validateStudioDesignCompatibility` checks the selected design against scene content.
- Template selection calls the validator before applying an infographic `designId` change.
- Preview and render also validate before provider renderer execution.

## Current Behavior Limitations Caused by Template 1/2

Observed limitations from the current template implementation:

- Some provider designs require data structures that not every scene can satisfy. The `g6-fishbone` failure shows that design selection is constrained by provider layout/data compatibility.
- The current foundation now rejects known incompatible design/content pairs before renderer execution.
- Behavior templates are easier than true scene templates in the current system because `SceneCompositionRecipe` can change camera, transitions, Board project, Board animation, and design ID, but it does not own a full scene-template schema.
- Board Template 2 composition changes reuse the same block IDs, block types, and block titles. Static layout changes are real, but object hierarchy and narrative content remain the same.
- Infographic Template 2 design changes preserve the same rows, nodes, and edges. Static provider composition changes are real when the design changes, but content does not change.

## G6 Template Failure Mode

Some G6 layouts require a specific graph structure beyond generic `nodes` and `edges`.

The observed failure `Tree structure not found for treeKey: tree` occurred when a generic graph scene was mapped to `g6-fishbone`. The `g6-fishbone` design uses AntV G6 fishbone layout. G6 now has a provider-native hierarchy adapter, but generic graph content is still not silently converted into hierarchy content.

Observed fact:

- `g6-fishbone` uses AntV G6 fishbone layout.
- Applying `g6-fishbone` to the generic `style-03-cause-map` data caused `Tree structure not found for treeKey: tree`.
- That failure was caused by a mismatch between the design layout requirement and the current generic graph data generated by `dataFrom`.
- `g6-fishbone` can now render when paired with explicit `StudioContent.providerData.kind === "g6-hierarchy"` content.
- The current safe Template 2 mapping falls `style-03-cause-map` back to Template 1 instead of using `g6-fishbone`.
- If `g6-fishbone` is paired with generic graph content through another path, compatibility validation rejects it before the G6 renderer constructs or renders an AntV `Graph`.

## True Scene Templates

For Template 2 to be a genuinely different scene template throughout the current architecture, the implementation would need current-code equivalents for all of the following responsibilities:

1. A persistent way to select a complete scene composition, not only a local UI recipe.

   Current state: `BehaviorTemplateId` and `SceneCompositionRecipe` live locally in `studio/src/AdvancedStudioApp.tsx`. They are not fields on `AdvancedStudioScene`.

2. A canonical owner for Board alternate layouts.

   Current state: Board layout is block geometry in `StudioProject.blocks`. Template 2 creates an alternate project through `createTemplate2BoardProject` in the UI file. No separate reusable Board scene-design registry was observed.

3. Provider-design compatibility rules.

   Current state: Infographic scene composition switches by `designId`. The G6 fishbone failure proves that not every registered design is compatible with every scene's generic nodes/edges data.

4. A design/content boundary that can safely vary visual form while preserving narrative content.

   Current state: G2/S2 generally consume `rows`; G6 consumes `nodes` and `edges`. Switching design IDs can preserve content, but compatibility depends on the target design factory and provider layout.

5. Template-owned static composition across all scene types.

   Current state: Template 2 changes static composition for 13 of 14 scenes. `style-03-cause-map` falls back to Template 1 because the attempted alternate G6 design was incompatible with the scene's data structure.

This section describes requirements implied by current code. It does not specify a new architecture.
