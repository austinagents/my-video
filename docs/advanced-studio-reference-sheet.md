# Advanced Studio Reference Sheet

## What Advanced Studio Is

Advanced Studio is a scene-based video editor built on Remotion. An authored
project is previewed in the Studio Player and sent through the same composition
for final rendering. Template 1 and Template 2 are complete projects that share
one architecture; they are not separate rendering systems.

## The Core Model

- **Project (`AdvancedStudioProject`)**: a title and an ordered list of scenes.
- **Scene (`AdvancedStudioScene`)**: a type, title, duration, content, camera,
  and optional crossfade.
- **Project duration**: the sum of all scene durations. Crossfades do not shorten
  the timeline.
- **Scene timing**: each scene starts when the previous scene ends.

## Core Providers and Owners

A **provider** is an installed external library that performs specialized work.
An **owner** is the provider or repository system responsible for a particular
behavior. AntV is a family of four separate visualization providers, not one
renderer.

| Provider or owner | What it does in Advanced Studio | What it does not own |
| --- | --- | --- |
| **React** | Runs the editor UI and component lifecycle | Video timing or visualization layout |
| **Vite** | Serves the local Studio app and its local render endpoint | Scene rendering or video encoding |
| **Remotion** | Evaluates the project one frame at a time and owns the composition used by preview and render | Chart, graph, table, or Board layout |
| **Remotion Player** | Plays that Remotion composition inside Advanced Studio and reports the current frame | A separate preview implementation |
| **Remotion CLI** | Executes the same composition for the final video file | Template authoring |
| **AntV G2** | Draws quantitative charts such as bars, lines, rankings, and KPIs | Scene timing, camera movement, or transitions |
| **AntV G6** | Draws graphs and executes network, tree, flow, and relationship layouts | Remotion frame timing or camera movement |
| **AntV S2** | Draws data tables, scorecards, and multidimensional sheets | Board blocks or charts |
| **AntV Infographic** | Generates the SVG content inside individual Board blocks | G2/G6/S2 scenes or Board positioning |
| **Advanced Studio** | Owns the project/scene contract, canonical timing, template selection, timeline UI, and render handoff | Low-level AntV drawing behavior |
| **Board** | Owns the spatial block surface, block placement, active block, and Board semantic motion | Project timing, camera paths, or transitions |
| **AntV design registry and adapters** | Select approved designs, validate compatible content, and convert project data into provider options | Provider drawing algorithms |
| **Camera paths and transitions** | Apply repository-defined Remotion transforms and crossfades around a completed scene | Provider-internal layout |

In short: **React runs the app, Advanced Studio decides what scene should be
shown, AntV or Board draws its content, and Remotion decides which video frame
is being evaluated and produces the final video.**

## The Four Scene Types

| Scene type | Best use | Repository route | Final drawing provider |
| --- | --- | --- | --- |
| **Board** | Spatial overview, chapter focus, process, comparison, summary | `BoardSceneRenderer` → `Board` → `AntVBlock` | Board layout plus AntV Infographic inside each block |
| **G2** | Charts, trends, rankings, KPIs, quantitative comparisons | `InfographicSceneRenderer` → `G2StudioRenderer` | AntV G2 |
| **G6** | Networks, flows, causes, trees, decision paths | `InfographicSceneRenderer` → `G6StudioRenderer` | AntV G6 |
| **S2** | Tables, scorecards, before/after metrics | `InfographicSceneRenderer` → `S2StudioRenderer` | AntV S2 |

Provider-first means a G2, G6, or S2 scene should use an existing registered
design and its content contract. Template code chooses the design and supplies
data; it does not recreate provider behavior. Board is repository-owned, but its
individual visual blocks are generated by AntV Infographic.

## How a Frame Is Rendered

`AdvancedStudioProject` → canonical scene timing → active scene → camera and
crossfade → Board or infographic renderer → Remotion Player or final render.

Preview and final render receive the same project. A preview/render difference
is therefore usually caused by readiness, provider layout behavior, or timing—not
by a separate preview composition.

## Board Authoring

A Board project controls its title, subtitle, blocks, and layout. Each block has
a type, title, position, size, and design preset. A Board scene selects an active
block and semantic motion such as focus, spotlight, reveal, count, or overview.

Normal Board generation currently supplies fixed sample labels and values for
each block type. `StudioBlock.syntax` is a supported escape hatch, but it should
only be considered when normal Board authoring cannot solve a proven content
conflict.

## Motion

- **Camera path** moves the complete scene; it does not own chart or Board
  semantics.
- **Board semantic motion** controls focus and emphasis inside the Board.
- **Provider design motion** reveals the infographic content.
- **Transition** is currently a crossfade between adjacent scenes.

When reporting a motion problem, identify whether the unwanted movement comes
from the camera, Board semantics, provider layout, or transition.

Provider lifecycle is also important: Remotion reevaluates React every frame,
but an AntV provider should render again only when its design, data, controls, or
dimensions change. Re-running an AntV layout every video frame can create
shaking even when the camera path is correct.

## Templates

- **Template 1** is the approved original baseline and must remain unchanged
  unless explicitly authorized.
- **Template 2** is a second standalone authored project. “Executive Signal” is
  its initial narrative, not a new architecture.

A new template should normally be data-only authoring: scenes, durations,
registered designs, provider content, Board layout, cameras, and crossfades.

## Useful Task Language

- **Content change**: copy, values, nodes, edges, rows, or Board titles.
- **Design selection**: choose a different registered G2/G6/S2 design.
- **Layout problem**: content is clipped, sparse, crowded, unstable, or poorly
  fitted to portrait/square/vertical.
- **Provider problem**: the registered design itself behaves incorrectly across
  projects.
- **Provider lifecycle problem**: AntV redraws or recalculates layout even though
  its design, data, controls, and dimensions did not change.
- **Board problem**: block placement, active-block behavior, generated block
  content, or Board semantic motion is wrong.
- **Studio UI problem**: selection, inspector, timeline, or local controls are
  wrong while the underlying composition remains correct.
- **Render handoff problem**: the correct project or format does not reach the
  Remotion CLI composition.
- **Camera problem**: the whole scene moves or scales incorrectly.
- **Timing problem**: scene length, start frame, project duration, or crossfade.
- **Parity problem**: the same frame differs between Player and final render.
- **Architecture change**: modifies a contract, renderer, provider, composition,
  registry structure, or render handoff. This requires evidence and approval.

For visual bugs, the most useful report is:
**template + scene title + frame + format + screenshot + expected behavior**.

## Default Development Rules

Preserve Template 1. Prefer authored data and registered providers. Strengthen
existing contracts before adding systems. Keep runtime changes minimal. Do not
fix unrelated technical debt automatically. Validate durations, provider
compatibility, Board references, Player behavior, final render, typecheck, build,
and `git diff --check`.
